這是《發現問題：AI 代理如何找到你的 WebMCP 工具？》的一篇朗讀稿，為語音合成做了優化，句子較短，方便自然停頓。

我們談論的主題是：WebMCP 的最大挑戰——工具如何被代理發現。

想像你的電商網站公開了工具：
“searchProducts”，可以搜尋商品；
“addToCart”，可以加入購物車；
“checkout”，可以結帳。

對人類來說，這些功能很有用。對於 AI 代理來說，如果能直接呼叫這些工具，很多任務可以更快完成。比起讓代理截圖、解析畫面，直接呼叫工具更有效率，也更可靠。

但問題在於：代理怎麼知道你的工具存在？

WebMCP 的提案中指出：目前沒有內建的機制，能在不造訪網站的情況下，讓代理預先知道哪個網站提供哪些工具。

今天的發現流程通常是這樣：
代理決定訪問網站。
瀏覽器載入頁面。
頁面執行 JavaScript，註冊工具。
代理才發現工具，然後呼叫工具。

這意味著，如果代理不知道要訪問哪些網站，就必須試探式地一一打開，這很慢，也浪費資源。

舉例來說，使用者說：「幫我找一件二百美元以下、適合夏季婚禮穿的紅色洋裝。」
代理要做的事很多：
它可能先查目錄服務，尋找哪些網站有服裝搜尋 API。或者，沒有目錄的情況下，代理必須逐一打開各家電商網站，等待頁面載入和 JavaScript 執行，然後才知道是否能呼叫工具。

每一次導航都會耗時。更糟的是，代理可能在很多不支援 WebMCP 的網站上浪費運算和代幣。

因此，發現問題是一個阻礙 WebMCP 廣泛採用的「最後一哩」難題。

接下來，我們討論四種可能的解法。

第一，聲明式清單。
網站在 Web App Manifest 或一個專門的檔案，比如「.well-known/webmcp.json」，聲明它提供的工具。
代理可以先抓取這個清單，再決定要不要導航到該站點。

這種方式的優點是快速、可被索引，適合建立搜尋目錄。
但也有缺點：靜態清單無法表達依賴使用者或狀態的工具。某些工具只有在特定情境下才可用，例如「undo」只在有可還原的編輯存在時才出現。

此外，公開列出所有 API 也可能引發隱私或商業資訊外洩的問題。最後，開發者需要在清單和程式碼之間保持同步，增加維護成本。

第二，PWA 與 Service Worker。
當使用者安裝一個 PWA 時，該應用的 service worker 可以註冊 WebMCP 工具，並在背景提供這些工具。

這讓工具在系統範圍內持久可用，無需每次打開頁面。代理可以在不用顯示介面的情況下，直接呼叫工具。

例如：安裝了某個日曆 PWA 之後，代理可以直接呼叫「addToCalendar」來新增行程，使用者僅在需要時被提示確認。

Service worker 解決了持久性問題，但它並未完全解決發現問題。代理仍須知道可以安裝哪個 PWA，或哪些站點的 PWA 提供所需工具。

第三，目錄與搜尋 API。
我們可以建立集中式或分散式的目錄。集中式服務像是應用商店，優點是統一索引和品質控制；缺點是集中化風險和可能的經濟扭曲。

分散式方案則像聯邦網路，網站自主管理 `/.well-known/webmcp.json`，任何人都能運行爬蟲或發現節點來聚合資料。代理可以查詢多個節點，取得結果並去重。

搜尋引擎也是一個自然的切入點。網站可以透過 Schema.org、OpenGraph 或 JSON-LD，將可用工具的資訊嵌入頁面，讓搜尋引擎擴展索引，為代理提供發現來源。

第四，AI 原生發現。
如果 WebMCP 的採用率足夠高，訓練資料中包含工具的描述，基礎模型可能「學會」哪些網站提供哪些功能。

在這種情況下，代理可憑藉模型的世界知識直接導航到合適的網站。但這依賴於訓練資料的完整性與時效性，且會面臨知識截止的問題。

那麼，開發者今天可以做些什麼來準備？

首先，新增語義化元資料。使用 OpenGraph 或 Schema.org，明確描述網站的功能點，例如搜尋、購物車與結帳。

其次，在網站上提供清晰的工具文件，例如建立 `/agents` 或 `/api-docs` 頁面，詳述工具的名稱、參數與限制。訓練中的代理會從這些內容學習。

第三，實驗 `/.well-known/webmcp.json` 端點，採用慣用的網域檔案位置，若未來標準採納此方式，能更容易被發現。

最後，參與 WebMCP 的社群討論，幫助形塑發現機制的標準。

展望未來，我預測會出現「代理 SEO」這個新職位。它的任務是針對 AI 代理進行優化：撰寫易於模型理解的工具描述、設計代理友善的元資料、分析代理呼叫行為，並持續優化工具的命名與說明。

總結來說，WebMCP 已經解決了效率問題。結構化工具比視覺自動化快且可靠。現在要解決的，是如何讓代理更快發現這些工具。

合適的解法可能是混合式的：靜態聲明加上已安裝的 PWA、搭配目錄索引，再由 LLM 加入背景知識。多重機制重疊，將讓代理在不同情境下選擇最佳路徑。

如果你今天就開始實作 WebMCP 工具，你就是在為未來的代理生態系統做準備。越早開始，越佔優勢。
